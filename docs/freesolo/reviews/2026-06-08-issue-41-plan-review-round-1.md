# Plan Review — Issue #41, Round 1

## Status
pass_with_findings

## Summary
The plan's module layout, TDD framing, and file structure faithfully mirror the spec and the `plan.ts`/`plan-command.test.ts` conventions established by #34. Tasks 1–2 include complete, copy-runnable failing tests and the type/error pattern is correct. However the plan has two architectural gaps — the in-process registry cannot serve cross-process `team stop` CLI invocations, and all of the critical spec edge cases (partial start failure, all-members-fail, monitor concurrency guard) are absent from test steps. Tasks 4–8 also revert to prose bullets without concrete test bodies, breaking the TDD discipline the plan itself establishes in Tasks 1–2. The gaps should be tightened before an implementing agent starts.

## Findings

1. **major — In-process registry is broken for cross-process `team stop` (plan: Task 8, Task 9 Step 3; spec: CLI Surface `team stop`).** The plan states the registry maps worktree paths to active managers "so `team stop` can tear down a team started in the same process." But `team start` explicitly does NOT block in the monitor loop — it exits after `create()`. When the user subsequently runs `freesolo team stop`, that is a separate OS process; the module-level registry map is empty. Task 9 Step 3 ("requires registered manager, calls tearDown") tests a path that will never fire in normal CLI usage. The spec already provides the correct approach: "`team stop` — Load manager state or read snapshot to confirm a team was started; `tearDown('cancelled')`." For a v1 CLI that creates but doesn't monitor, `team stop` should reconstruct adapter references from the snapshot (or accept that it can only stop already-running adapters by calling `adapter.stop()` directly). Either way, the registry must be dropped or reduced to a development/test convenience, and Task 9 Step 3 must be rewritten to test the snapshot-driven stop path. As written, the CLI stop command will silently find no manager and do nothing.

2. **major — Partial start failure and all-fail cases are unplanned (plan: Task 6 Step 1, Task 7; spec: `create(definition)` — Start failures).** The spec dedicates a paragraph to this: (a) if a member fails to start, mark it `error`, emit `agent.stopped` with `{ reason: 'start-failed' }`, and continue starting others; (b) if **all** members fail, transition `tearing-down → stopped(error)` without entering `running`. Neither case appears in any test step. Add at minimum two tests to `team-lifecycle-manager.test.ts`: one scripted factory where one of two adapters throws on `start()` (assert phase `running`, one member `error`, `agent.stopped` event with `reason: 'start-failed'`), and one where all adapters throw (assert phase `stopped`, `stopReason: 'error'`, no `team.created` event).

3. **major — Monitor concurrency guard is unplanned (plan: Task 7; spec: `monitor()` — "concurrent calls throw `invalid-state`").** The spec explicitly requires that a second `monitor()` call on the same manager throws `TeamLifecycleError('invalid-state')`. This is not listed in any Task 7 test step. Add a test that calls `monitor()` twice concurrently and asserts the second rejects with `{ name: 'TeamLifecycleError', code: 'invalid-state' }`.

4. **major — Tasks 4–8 revert to prose; TDD discipline breaks after Task 2 (plan: Tasks 4–8 Step 1 blocks).** Tasks 1–2 have complete failing-test code with imports, describe blocks, and assertions. Tasks 4–8 describe tests only in prose bullets ("Test round-trip write/read…", "Write failing create test — Use in-memory EventLog mock…", "Write failing monitor completion test — Scripted adapters that transition to stopped…"). An implementing agent must invent the entire test scaffold from scratch for six tasks. Provide concrete test stubs matching the Task 1–2 style for at least the following: Task 4 store round-trip, Task 5 `isMemberInactive` + `isTeamComplete` helpers, Task 6 `create()` happy path (event payload assertions), Task 7 cancel and completion paths.

5. **minor — `TeamMemberRuntime` is absent from the `types.ts` File Structure entry (plan: File Structure table; spec: Data Model).** The spec defines `TeamMemberRuntime` — the interface holding `spec`, `adapter`, `status`, `blockedAt`, and `blockedReason` that the manager stores in its member map at runtime. The File Structure lists `TeamPhase`, `TeamMemberSpec`, `TeamRuntimeSnapshot`, `TeamLifecycleConfig`, `TeamStopReason` but omits `TeamMemberRuntime`. Without it in Task 1, an implementing agent will not create it until Task 6 and will miss the `adapter: AgentAdapter` import. Add `TeamMemberRuntime` to the Task 1 file description and include it in `src/teams/index.ts` barrel.

6. **minor — `AgentHost` vs `PlannerHost` naming divergence is unacknowledged (plan: Task 1 types, Task 2 test; spec: `import type { AgentHost } from '../planner/types.js'`; codebase: `PlannerHost` in `src/planner/schemas/team-definition.ts`).** The spec's data model imports `AgentHost` from `../planner/types.js`, but the current worktree exports `PlannerHost` from `src/planner/schemas/team-definition.ts`. The spec's Dependency Note flags that the spec is written against merged interfaces not yet in this worktree. The plan silently uses the current path and name without alerting the implementer. Add a note in Task 1 Step 3: use `PlannerHost` (current worktree) and alias it `AgentHost` in `src/teams/types.ts` if the merged branch renames it, so the types match the spec without churn.

7. **minor — `TeamCommandDeps` interface is not specified (plan: Task 9 Step 5 "follow plan.ts patterns"; reference: `plan.ts` PlanCommandDeps has 16 fields).** `PlanCommandDeps` took significant effort to define correctly; the equivalent for `team.ts` needs at minimum: `resolveRepoRoot`, `resolveRepoRef`, `resolveIssueNumber`, `readState`, `writeState`, `readTeamPlan`, `createTeamManager` (factory returning `TeamLifecycleManager`), `readTeamRuntimeSnapshot`, `env`, `write`, `setExitCode`. Without the interface sketched in advance, Task 9 Step 5 implementation will invent inconsistent field names that break the injection pattern. Add the interface outline to Task 9's setup.

8. **minor — Task 4 store test omits `git init` (plan: Task 4 Step 1; prior review finding: issue-34 round-1 finding #3).** `getFreesoloPath` calls `execa('git', ['rev-parse', '--git-path', 'freesolo'], …)`. A bare `mkdtemp` directory is not a git repo; the store test will throw from `git rev-parse` rather than pass. Mirror the `makeRepo()` helper pattern from `verification-store.test.ts` or `issue-id.test.ts` (run `git init --quiet` after `mkdtemp`). This exact issue was flagged in the issue-34 round-1 review.

9. **minor — `buildMemberPrompt` output is unspecified and untested (plan: Task 6 Step 2 "Implement prompt.ts"; spec: `buildMemberPrompt(member)` — includes role name, responsibility, issue number).** The spec states the prompt must include role name, responsibility, and issue number so the agent knows its context. There is no test planned for `buildMemberPrompt` output, and no acceptance criteria for prompt content. Add a simple snapshot/assertion test in `team-lifecycle-manager.test.ts` or a separate `team-prompt.test.ts` that checks the returned string contains `member.roleName`, `member.responsibility`, and the issue number.

10. **nit — Task 7 Step 3 inactivity test does not explain how fake time is injected (plan: Task 7 Step 3).** The `TeamLifecycleManagerDeps` has `now?: () => Date` and `sleep?: (ms: number) => Promise<void>` for controllable time. Task 7 Step 3 says "adapter with old `lastActivityAt` and short `memberBlockedTimeoutMs`" but does not show how `now` is set to a future timestamp to make `lastActivityAt` appear stale. Show the `now` dep assignment in the failing test stub, e.g., `now: () => new Date(Date.now() + 2 * memberBlockedTimeoutMs)`.

11. **nit — Event-log-types test count claim should be verified (plan: Task 3 Step 1 "Twelve event types").** Current `EVENT_TYPES` in `src/event-log/types.ts` has 8 entries; adding 4 team types yields 12. The count is correct, but the plan should list the expected 12 types explicitly in the test expectation rather than just saying "twelve" — avoids silent pass if the implementer accidentally overwrites an existing type.

## What Looks Good

- The `src/teams/` module layout mirrors `src/planner/` and `src/verification/` exactly — separate domain logic from CLI wiring, consistent with ADR-0002 and the established pattern.
- Tasks 1–2 are fully TDD-correct: complete failing test files with imports and assertions, explicit "Expected: FAIL" / "Expected: PASS" run steps, and concrete implementations.
- `errors.ts` is correctly split out as a dedicated file; the `TeamLifecycleError` pattern matches `AgentAdapterError` and `EventLogError` in the codebase.
- The decision to add `src/teams/prompt.ts` as a pure, separately testable function matches how `buildMemberPrompt` is described in the spec — pure function, no side effects.
- Event count arithmetic is correct: 8 existing + 4 new = 12; the test modification step calls this out explicitly.
- Task 3 Step 4 lists all six builder functions (`buildTeamCreatedEvent`, `buildAgentCreatedEvent`, `buildTeamMemberBlockedEvent`, `buildTeamTearingDownEvent`, `buildTeamTornDownEvent`, `buildAgentStoppedEvent`) — complete coverage of every event emitted by the manager.
- `AgentAdapterFactory` stays in `manager.ts` per spec — keeps the factory interface co-located with its consumer, avoids circular imports.
- Engine gate is correctly placed only on `team start` (state-mutating), not `team status` — matches the pattern in `plan generate` and `plan approve`.
- Task 10 closes with `npm test` and `npm run build` gates, consistent with all prior plan patterns in this repo.
- The decision to use `ScriptedAgentAdapter` via factory injection makes all manager tests fully deterministic without subprocesses — correct for v1.
