# Plan Review — Issue #28, Round 1

## Status
pass_with_findings

## Summary
The plan covers all four issue acceptance criteria in its self-review table, mirrors the spec's `src/state/` layout and schema, and follows the repo's established DI style (`StartPlanDeps`, `registerStateCommands`-style command registration). Tasks 1–2 are exemplary red/green TDD with copy-runnable code. The main gaps are operational: the migration runner reads adjacent `.sql` files but the build pipeline only emits TypeScript to `dist/`, so production `freesolo` would fail to migrate; `freesolo start` integration omits the spec's fatal-error contract and precise upsert ordering relative to `session.json`; and several CLI/testing details from the spec (`--json`, drift exit code `2`, drift row-selection orchestration) are underspecified. None are fundamental architecture disagreements — amend the plan before implementation and it should land cleanly.

## Findings

1. **major — SQL migration files will not exist beside compiled output in `dist/` (plan: Task 3 Step 3, Task 9 Step 2).** The spec and plan both place migrations at `src/state/migrations/001_worktrees.sql` and load them via `import.meta.url`. `tsconfig.json` includes only `src/**/*.ts`; `npm run build` emits JS to `dist/` but does not copy `.sql` assets. The published bin is `./dist/src/bin.js`, so a built `freesolo start` or `freesolo worktrees list` would open the DB then fail to find migration files at runtime. Tests run against source via Vitest imports and would not catch this — Task 9's `npm run build` gate compiles TypeScript only. Fix before coding: extend the build script to copy `src/state/migrations/*.sql` → `dist/src/state/migrations/`, or resolve migration paths from a known package root; add a post-build smoke test (or integration test invoking compiled output) that calls `runMigrations` successfully.

2. **major — `freesolo start` metadata upsert failure is not specified as fatal (plan: Task 8; spec: Integration § `freesolo start`).** The spec states: "Failure to persist metadata is **fatal** for `start` — the user sees a clear error and the host is not launched." Task 8 adds `upsertWorktreeMetadata` to deps and asserts the happy-path call, but does not specify that `StateDbError` (or any upsert failure) must abort before host launch with a clear stderr message and non-zero exit. `startAction` today only catches worktree-specific errors; an unhandled DB error would throw or behave inconsistently. Add: (a) explicit placement in Task 8 that upsert errors propagate to `startAction` with `process.exitCode = 1` and no adapter launch; (b) an integration test where `upsertWorktreeMetadata` rejects and asserts the plan does not reach launch / session side effects beyond the upsert point.

3. **major — Upsert timing relative to `session.json` is vague (plan: Task 8 Step 2; spec: Integration § `freesolo start`).** The spec requires upsert "after `worktreePath` is resolved and **before writing `session.json`**" so metadata failure does not leave a session file without a DB row (and vice versa). Task 8 Step 2 says only "Call after `worktreePath` resolved, only when `!input.printOnly`." In `start.ts` today, `writeSessionState` runs at lines 350–378 — an implementer following the plan literally could upsert after session writes. Tighten Step 2 to: call `upsertWorktreeMetadata` immediately after `worktreePath` / `shouldRunSetup` resolution and **before** `writeSessionState` / `writeIssuePacket`; add an ordering assertion in the integration test (e.g. mock records call order: upsert before session write).

4. **minor — Drift command exit code `2` for operational errors is missing (plan: Task 7 Step 2; spec: CLI Surface).** The spec defines exit `0` = no drift, `1` = drift found, `2` = operational error (DB open/migrate failure, git resolution failure). Task 7 tests only exit `0` and `1`. Add a test case where `getWorktreeStore()` or `resolveRepoRoot` throws and assert `setExitCode(2)` (or equivalent), matching the `state.ts` pattern of injected `setExitCode`.

5. **minor — `--json` flags on `list` and `drift` are in the spec but absent from plan tasks and tests (plan: Task 7; spec: CLI Surface).** The spec exposes `freesolo worktrees list [--json]` and `freesolo worktrees drift [--json]`. Task 7 describes human-readable table output only. Add `--json` option registration, JSON serialization of rows / `WorktreeDriftReport`, and at least one test per subcommand asserting structured output on stdout (mirror patterns elsewhere if any; otherwise follow Commander option + deps.write testing from `state-command.test.ts`).

6. **minor — Drift command orchestration (DB row selection) is underspecified (plan: Task 5 + Task 7; spec: Drift detection).** The pure `detectWorktreeDrift` function is well scoped, but `driftAction` must also decide which DB rows to pass in: the spec says load rows whose `path` is in the repo's `git worktree list` set **or** whose path no longer exists on disk (`fs.access`). Task 7 Step 3 says "Default deps call `getWorktreeStore()`, `listWorktreeEntries`, `resolveRepoRoot`" without describing the filter/join step. Without this, an implementer may pass `store.list()` (all host-global rows) into the pure function and mis-classify rows from other repos. Add a short orchestration note or helper (e.g. `loadDriftCandidates(store, gitPaths, pathExists)`) and a command-level test with fixture rows inside/outside the git path set.

7. **minor — "Metadata survives restart" lacks an explicit round-trip test (plan: Task 4 + Task 6; spec: Testing Strategy, Acceptance Criteria).** The spec calls for "upsert → reopen → read". Task 4 uses `:memory:` DBs (no reopen), and Task 6's smoke test is described in one line without concrete steps. Task 3 covers migration re-run idempotency on the same handle, not a closed-and-reopened file DB. Add a test in Task 4 or Task 6: upsert on a temp-file DB via `FREESOLO_STATE_DIR`, close, reopen with `getWorktreeStore()`, assert `getByPath` returns the row.

8. **minor — Tasks 3–5 TDD steps are less concrete than Tasks 1–2 (plan: Tasks 3, 4, 5).** Steps say "Write failing migration test" / "Write failing tests for upsert…" without full test code blocks, unlike Task 1–2 which are copy-runnable. Not TBD placeholders, but increases implementer variance. Recommend pasting at least skeleton test files (as Task 1 does) for migration re-run, upsert `created_at` preservation, and drift `onDiskOnly` / `metadataOnly` cases.

9. **minor — `better-sqlite3` native dependency implications not acknowledged (plan: Task 2, Task 9).** `better-sqlite3` compiles native bindings via `node-gyp` on `npm install`. The plan adds the dependency but does not note platform/build-tool requirements (Python, make, Xcode CLT on macOS, MSVC on Windows) or that CI / contributor onboarding must succeed with native compilation. Task 9 should include a clean `npm ci && npm test && npm run build` note after adding the dep, and README/contributor docs if install failures are likely. Acceptable for v1, but worth one plan sentence so implementers aren't surprised.

10. **minor — Worktrees command DI sketch is incomplete vs `state.ts` (plan: Task 7 Step 3).** The plan says "Follow DI pattern from `src/commands/state.ts`" but does not define a `WorktreesCommandDeps` interface with injectable `write`, `setExitCode`, and store/git helpers. `state-command.test.ts` captures stdout/stderr and exit codes via deps; without the same interface, command tests may coupling to `process.stdout` / `process.exitCode`. Specify `registerWorktreesCommands(program, deps = defaultDeps)` with the same IO injection shape.

11. **nit — `tests/unit/state-db.test.ts` created in Task 2 but omitted from the File Structure list (plan: top § File Structure vs Task 2).** Minor documentation inconsistency; add to the file list for agent navigability.

12. **nit — Self-review row "Idempotent concurrent upsert" cites Task 2 (WAL) + Task 4 (ON CONFLICT) but no concurrent or double-process test (plan: Self-Review Checklist).** The spec mentions concurrent processes; WAL + upsert SQL is sufficient for v1, but the checklist overclaims "tested" — only WAL pragma and ON CONFLICT unit tests are planned. Reword to "implemented" or add a lightweight test that two sequential upserts on the same path leave one row.

13. **nit — No per-task commit guidance (plan: overall structure).** Other FreeSolo plans (e.g. #33, #20) include commit steps per task. This plan ends at verification only. Optional for agent workflow consistency, not a spec gap.

## What looks good

- All four issue acceptance criteria and spec requirements appear in the self-review checklist with sensible task mappings; file layout (`src/state/{paths,db,migrations,worktrees,drift,index}.ts`, `src/commands/worktrees.ts`) matches the spec and ADR-0001.
- Tasks 1–2 demonstrate strict red/green TDD with exact test commands, expected failure modes, and complete implementation snippets; `FREESOLO_STATE_DIR` override matches the spec's test-only path strategy.
- `WorktreeStore` receives an injected `Database` (spec DI requirement); `getWorktreeStore()` factory for production matches the spec's "open once per invocation" model.
- Schema, upsert SQL (`ON CONFLICT` preserving `created_at`), WAL + `busy_timeout` pragmas, error types (`StateDbError`, `WorktreeNotFoundError`), and `StartPlanDeps.upsertWorktreeMetadata` hook align with the spec character-for-character where specified.
- `--print-only` correctly skips metadata writes (Task 8 Step 2); drift pure function's injectable `pathExists` is a sound testability extension of the spec's `fs.access` note.
- Test files follow existing flat `tests/unit/<name>.test.ts` convention (unlike the #33 subdirectory deviation); integration touchpoint `tests/integration/start-command.test.ts` matches how `start` is tested today.
- Task 9 full-suite + build gate will catch TypeScript regressions across the existing 33 test files.
