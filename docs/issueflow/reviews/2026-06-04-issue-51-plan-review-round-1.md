# Plan Review Round 1 — Issue #51

## Verdict
pass_with_findings

## Summary
The plan delivers every spec acceptance criterion with concrete code, tests, and commits. The module layout matches the spec verbatim, identifier names are character-perfect, the TDD red/green loop is followed for every task that ships code, and the engine-isolation regex (despite the name collision with `src/workflow/state-store.ts`) actually distinguishes the legitimate workflow imports from the new module imports. The findings below are all fixable nits/should-fixes: a test that asserts nothing about WAL/SHM siblings under `safeDelete`, a brittle first-spec assertion in the same file, an IPC-flush race in the concurrency worker fixture, two typing risks from `Function.prototype.bind` against better-sqlite3's overloaded methods, and a handful of count / wording slips.

## Findings

1. **should-fix — Task 8, Step 1, second spec ("also moves -wal and -shm when present") doesn't actually verify WAL/SHM moves (`tests/unit/state-store-safe-delete.test.ts`).** The loop captures `sourceExisted` by `fs.stat`-ing the source path *after* `safeDelete` has already moved files away, so `sourceExisted` is always `false` (file moved or never existed). The `if (sourceExisted) { ... }` branch never runs, and the trash-copy assertion at `path.join(trashRoot, sibling)` is never executed. The test will pass even if `safeDelete` silently drops `-wal`/`-shm`. Fix: snapshot pre-existence *before* calling `safeDelete`, then after the call assert (a) the source is gone (which it always is, because the file was either moved or never existed), and (b) for every sibling that pre-existed, `fs.stat(path.join(trashRoot, sibling))` resolves and `result.movedFiles` contains it.

2. **should-fix — Task 8, Step 1, first spec uses a strict-equality assertion against a list that depends on SQLite's checkpoint behaviour at close (`tests/unit/state-store-safe-delete.test.ts`).** `expect(result.movedFiles.sort()).toEqual(['state.db'].sort())` will fail (or flake across platforms / `better-sqlite3` versions) if SQLite leaves a `-wal` or `-shm` file after `db.close()`. The fixture closes the seed DB with the default checkpoint behaviour, so on most platforms WAL is consolidated and removed — but this is not contractually guaranteed. Either (a) explicitly checkpoint-then-truncate (`db.pragma('wal_checkpoint(TRUNCATE)')` then `db.close()`) and assert the strict list, or (b) relax the assertion to `expect(result.movedFiles).toContain('state.db')` and let the second spec own the WAL/SHM cases.

3. **should-fix — Task 11, worker fixture has an IPC-flush race (`tests/fixtures/state-store-concurrent-writer.mjs`).** The sequence `process.send?.({ type: 'done', ... }); process.exit(0);` calls `process.exit` immediately after queueing the IPC message; Node does not guarantee the pending IPC channel is flushed before exit, so the parent occasionally sees `code === 0` but never receives the `done` message — and the parent then rejects with `worker exited with code 0 (report=undefined)`. Use the callback form: `process.send({ type: 'done', pid: process.pid, written: rowCount }, undefined, undefined, () => process.exit(0));` (or `await new Promise<void>((r) => process.send({...}, () => r()))` and then exit). Without this fix the test is a likely intermittent failure under load.

4. **should-fix — `prepare: db.prepare.bind(db)` and `pragma: db.pragma.bind(db)` (Task 9, `src/state-store/store.ts`).** `better-sqlite3`'s `prepare` is a generic overload (`prepare<P, R>(sql: string): Statement<P, R>`); `Function.prototype.bind` typically collapses overloads/generics to the last-resort call signature, and assigning the bound result to `Database.Database['prepare']` (which is the generic signature) can fail under `strict: true` with TS 5.9. Even if it compiles today, downstream consumers calling `store.prepare<MyParams, MyRow>('...')` may lose the generic arguments. Switch to a passthrough that preserves the call signature: `prepare: ((sql: string) => db.prepare(sql)) as Database.Database['prepare']` or, better, declare them as methods (`prepare(sql) { return db.prepare(sql); }`) and adjust the `StateStore` interface to use method-signature syntax. Same treatment for `pragma`. (Side benefit: `prepare`/`pragma` then become positioned to call `ensureOpen()` the way `exec`/`transaction` do — they currently bypass the open-check, which is inconsistent with the rest of the surface and silently lets callers prepare against a closed DB.)

5. **should-fix — Engine-isolation regex name is the same as the legitimate workflow file `src/workflow/state-store.ts` (Task 10, `tests/unit/state-store-engine-isolation.test.ts`).** The regex is correctly constructed — it requires either a closing quote or a `/` immediately after `state-store`, so `from './state-store.js'` does *not* match (verified against the current `src/workflow/engine.ts` and `src/workflow/policy.ts`). However, if any workflow file ever migrates to `from './state-store'` (no extension) or `from './state-store/foo.js'` (after a refactor that turns the file into a folder), the regex will start producing false positives that the plan reader will struggle to debug. Add a code comment to the test explaining what the regex matches and what it deliberately doesn't (`state-store.js` file vs. `state-store/` directory), mirroring the runner-isolation test's terse comment style.

6. **nit — Task 2 reports "PASS (4 specs)" but the test file has 6 `it` blocks (`tests/unit/state-store-types.test.ts`).** Two in `describe('StateStoreError', …)` + four in `describe('state-store types (structural)', …)` = 6. Update the Step 2/Step 4 "Expected" lines to "6 specs".

7. **nit — Task 1, Step 3 leaves a smoke-test file in `/tmp` until the executor remembers to delete it.** The plan says "Then delete the file: `rm /tmp/sqlite-smoke.mjs`." This is fine, but the file lives outside the repo so a forgotten cleanup leaves litter. Cheap improvement: use `mktemp` (`SMOKE=$(mktemp --suffix=.mjs)`) so the executor doesn't have to remember the path.

8. **nit — Task 12, Step 3 expects `ls dist/src/state-store/` to list `migrations/001-init.js` and `migrations/index.js`.** `ls` without `-R` shows `migrations` as a single directory entry, not its contents. Either change the command to `ls -R dist/src/state-store/` or rewrite the expectation to "expect a `migrations/` subdirectory plus the seven sibling `.js` files".

9. **nit — Task 9, `src/state-store/store.ts` line `pragma: db.pragma.bind(db)` (and `prepare`) bypass `ensureOpen()`.** All other surface methods (`exec`, `transaction`, `backup`, `safeDelete`) check the closed flag; `prepare` and `pragma` do not, so a `store.close(); store.prepare('SELECT 1');` returns a Statement against a closed DB which then throws a raw better-sqlite3 error on `.run()`/`.get()`. This is a minor consistency wart, easy to fix when Finding 4 lands. Optional: add a one-line test in `state-store-api.test.ts` that pins `prepare` after `close()` throws `StateStoreError('closed', ...)`.

10. **nit — Spec line numbering in the "Acceptance-criteria coverage" table at the bottom of the plan refers to a "Self-imposed" criterion (workflow-engine isolation) that isn't in the GitHub issue's acceptance list.** It's a real and valuable regression test, but flag it consistently in the plan body (e.g., "extra invariant we choose to lock") so a reviewer skimming the table doesn't try to find it in the upstream issue.

- **Migration runner design is sound and complete**: pre-validates duplicates before any DB write, bootstraps `schema_migrations`, detects "applied but unknown to caller" (typed as `migration-version-conflict` with a helpful message), and wraps each `up` in `db.transaction(...)` so a failing migration rolls back its DDL plus its own schema_migrations insert. Insertion of the `schema_migrations` row happens *inside* the same transaction as the `up`, which is the right place — Round-1 finding for issue #20 made the same call for verification artefacts.
- **TDD discipline is strict across every task that ships code.** Each task has a failing-test step with the exact expected failure message ("module … cannot be resolved" / "is not exported"), the minimum implementation that makes it pass, a passing-test verification, then a commit. Task 6 (the only no-test task) correctly substitutes `tsc --noEmit` for the red/green loop — appropriate for adding a no-op default-export plus a const.
- **Concurrent-writer test exercises the real spec criterion.** Forking 4 workers × 50 rows each, with independent `better-sqlite3` connections on a shared WAL-mode DB, then asserting (a) total row count, (b) per-PID counts, (c) global id monotonicity, (d) no worker exited non-zero. Any `SQLITE_BUSY` propagation past `busy_timeout = 5000` would surface as a worker exit code, which the parent rejects on. That's the right level for a "two IssueFlow processes appending events" regression test, and it lives under `tests/unit/` per the spec's explicit decision.
- **Engine-isolation regex correctly threads the `state-store` filename collision.** The regex requires either a closing quote or a `/` immediately after `state-store`, so `from './state-store.js'` (the existing workflow file) does not match, while `from '../state-store/types.js'` or `from '../state-store'` (the new module) do. No leaky abstractions — `prepare` and `pragma` are passed through to better-sqlite3 as the spec promised.
- **Failure modes are typed end-to-end.** Every catch wraps the underlying SQLite/Node error in a `StateStoreError` with one of the documented codes. `open-failed`, `migration-failed`, `migration-version-conflict`, `backup-failed`, `safe-delete-failed`, `closed` all have direct test coverage.

- **Dependency footprint is one runtime + one type package**, exactly as the spec called out; no incidental dev tooling, no build-step changes (TypeScript-only migrations sidestep the "tsc doesn't copy non-`.ts` files" trap the spec flagged).
- **`tsconfig.json` compatibility verified by inspection** — `strict: true`, `module: NodeNext`, `target: ES2022`, `noEmitOnError: true`. The plan's `.js` import extensions, `import type` usage, no `any`, and method-vs-property choices line up with the existing modules (`src/runners/`, `src/agents/`).
